# AI Work Judge 技术架构

## 当前目标架构

当前项目准备从“完整 AI 审计工作台”收缩为“浏览器插件 + 极简后端记录 + 轻量 Web 总后台”。

核心结构分成三层：

- `浏览器插件 Popup`
  - 主入口
  - 读取当前页面标题和 URL
  - 获取 AI 建议
  - 人工确认项目状态
  - 提交备注并保存
- `Next.js API`
  - 保存项目记录
  - 保存标签历史
  - 提供项目列表
  - 提供 AI 建议接口
- `Web 总后台`
  - 只显示项目列表、状态筛选、最近更新和待推进项目

## 共享类型

新的主数据模型围绕 `ProjectRecord` 展开，而不是围绕长报告。

```ts
type ProjectStatus = "消费" | "转化中" | "生产" | "暂停";

type ProjectRecord = {
  id: string;
  title: string;
  url: string;
  source: "web" | "github" | "notion" | "docs" | "unknown";
  currentStatus: ProjectStatus;
  createdAt: string;
  updatedAt: string;
  lastNote: string;
  aiSuggestion?: "更像消费" | "更像转化中" | "更像生产" | "建议人工判断";
  tags: ProjectTagRecord[];
};

type ProjectTagRecord = {
  id: string;
  projectId: string;
  createdAt: string;
  humanStatus: ProjectStatus;
  aiSuggestion?: "更像消费" | "更像转化中" | "更像生产" | "建议人工判断";
  note: string;
};
```

## API 方向

后端只保留 4 类能力：

- `GET /api/projects`
  - 获取全部项目
- `POST /api/projects`
  - 新建项目
- `POST /api/project-tags`
  - 为项目追加一次标签记录
- `POST /api/project-ai-suggestion`
  - 获取 AI 建议

旧的长流程接口：

- `/api/analyze-link`
- `/api/analyze-artifact`
- `/api/deliverability-report`
- `/api/weekly-summary`

不再作为主入口，只保留为过渡代码，后续逐步废弃。

## AI 的职责

AI 不再负责生成长篇报告，也不负责最终判定。

AI 现在只负责：

- 根据标题、URL、来源和备注生成一个轻量建议
- 输出一个简短理由

最终状态始终来自人工标签。

## 存储策略

第一版后端可先使用本地 JSON 文件仓储或轻量存储抽象，不强制数据库。

目标数据只需要：

- 项目基础信息
- 当前状态
- 最近备注
- 标签历史
- 最近一次 AI 建议

## Web 总后台职责

Web 页面不是完整工作台，而是极简后台：

- 展示全部项目
- 按状态筛选
- 看最近更新
- 看哪些项目仍处于 `消费` 或 `转化中`

## 插件 Popup 职责

插件是主入口，界面只保留：

- 当前页面标题
- 当前页面 URL
- AI 建议
- 状态按钮
- 一句备注
- 保存按钮

## DeepSeek 配置

`.env.example` 仍然保留：

```env
DEEPSEEK_API_KEY=
DEEPSEEK_API_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
```

`.env.local` 由用户自行维护，不提交、不打印。

## 验证清单

- `npm run typecheck`
- `npm run build`
- 插件能读取当前 tab 的标题和 URL
- 插件能请求 AI 建议
- 插件能提交人工标签
- Web 总后台能看到所有项目
- 能按 `消费 / 转化中 / 生产 / 暂停` 筛选
